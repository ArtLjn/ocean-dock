# Session Handoff Skill

将上一轮 Claude Code session summary 解析为结构化交接文档，帮助新会话快速理解上下文并无缝继续工作。

## 解析流程

### 第 1 步：识别并提取信息

从用户粘贴的 session summary 中提取以下信息块：

**1. 项目元信息**
- 项目路径和 Git 分支
- Session ID（如果有）

**2. 变更清单**
- 修改的文件列表及编辑次数（按编辑次数降序排列，标注高改动文件 ≥10 次）
- 新建的文件列表
- 工作量统计（编辑次数、文件读取次数、命令执行次数）

**3. 需求时间线**
- 从"用户需求记录"中提取每条需求
- 分析需求之间的逻辑关系（顺序依赖 / 独立 / 方向变更）
- 标注需求的当前状态：已完成 / 进行中 / 被搁置 / 方向变更

**4. 关键技术分析**
- 从"关键分析"中提取技术发现、问题根因、解决方案
- 识别重复出现的问题模式

**5. 代码上下文（按需）**
- 对于高改动文件（编辑 ≥10 次），读取当前文件内容
- 理解这些文件的当前状态和核心逻辑
- 注意：只在用户确认需要深入了解时才读取文件，不要一开始就大量读取

### 第 2 步：分析需求状态

对每条用户需求进行状态判定：

| 状态 | 判定标准 |
|------|----------|
| 已完成 | 有对应的代码修改，且后续需求不再提及 |
| 进行中 | 最近几条需求都在围绕该主题 |
| 被搁置 | 用户明确说"先继续下一步"或转向其他话题 |
| 方向变更 | 用户说"转个方向"或"不如试试" |

### 第 3 步：生成结构化交接文档

使用以下模板输出：

```markdown
# Session 交接报告

> Session ID: [id]
> 项目: [路径] | 分支: [分支]
> 生成时间: [当前时间]

---

## 一、工作概览

**工作量**: [N] 次代码编辑, [N] 次文件读取, [N] 次命令执行
**修改文件**: [N] 个 | **新建文件**: [N] 个
**核心改动文件**（编辑 ≥10 次）:
- `path/to/file` ([N] 次编辑) - [一句话说明当前改动重点]

---

## 二、需求时间线与状态

| # | 需求 | 状态 | 备注 |
|---|------|------|------|
| 1 | [需求摘要] | 已完成 | - |
| 2 | [需求摘要] | 已完成 | - |
| 3 | [需求摘要] | 被搁置 | 用户说"先继续下一步" |
| ... | ... | ... | ... |

**当前活跃方向**: [最近几条需求围绕的主题]

---

## 三、关键技术发现

### [发现 1 标题]
- **问题**: [描述]
- **根因**: [描述]
- **解决/进展**: [描述]

### [发现 2 标题]
...

---

## 四、待办事项（从需求记录推导）

- [ ] [从"进行中"需求推导出的下一步]
- [ ] [从"被搁置"需求推导出的回溯项]
- [ ] [从关键分析中推导出的优化项]

---

## 五、建议的下一步

基于需求时间线和技术分析，建议：

1. **最优先**: [最近活跃方向的具体下一步]
2. **可选回溯**: [被搁置但可能有价值的任务]
3. **长期优化**: [从分析中发现的可改进项]
```

### 第 4 步：交互确认

生成文档后：

1. 展示完整交接报告
2. 询问用户："以上理解是否准确？你想从哪里继续？"
3. 根据用户选择，深入相关代码文件并开始工作

## 关键原则

- **不要臆测**：对于不确定的需求状态，标注"待确认"而非猜测
- **聚焦变更**：高编辑次数的文件是重点，低编辑次数的文件一笔带过
- **识别模式**：如果用户多次测试同一功能（如多次评测），说明这是当前重点
- **保持简洁**：交接文档控制在 200 行以内，细节留给后续按需深入
- **始终使用中文**回复
