# 日报生成（成果导向版）

根据今天所有 Claude Code session 的工作记录，生成一份**以成果和价值为导向**的工作日报 docx 文件。

## 核心原则

- **说"做了什么"，不说"改了哪些文件"**：日报是给人看的，不是 git log
- **有指标就带上**：准确率、耗时、数量等数字是最好的价值证明，但没指标就别硬凑
- **自然真实**：用正常的工作语言描述，不要堆砌"赋能"、"落地"、"体系建设"等空泛词汇
- **适度包装**：把"修了个 bug"说成"优化了 XX 模块的稳定性"就行，别过度美化
- **少暴露过程细节**：不需要写排查过程、踩坑经历，只写结果
- 文件变更不要出现在正文中
- 不要编造数据中没有的量化指标

## 执行步骤

### 第一步：运行数据采集脚本

脚本会自动从 claude_manager 读取今天的 session 数据，按项目过滤，输出结构化 JSON。

```bash
python3 <skill_dir>/scripts/collect_data.py [项目名1] [项目名2] ...
```

- 如果用户指定了项目名（如 `/daily-report airQA skill_engine`），只采集这些项目的 session
- 如果用户没有指定项目名，采集今天所有项目的 session
- 将输出保存到临时文件 `/tmp/daily_report_data.json`

### 第二步：润色工作描述

读取采集到的 JSON 数据，对每个项目的每条 `main_task` 进行润色：

1. **去噪音**：去掉 `@文件路径`、代码片段、命令行输出等非人类可读内容
2. **功能化**：将"修改了 XX 文件"改写为"实现了 XX 功能"
3. **带上指标**：如果 session 中有 `metrics`（量化指标），自然地融入描述中
4. **适度包装**：用正常的职业用语描述工作，不过度美化也不自贬
5. **合并同类**：同一项目的多个 session 如果在做同一件事，合并为一条
6. **控制长度**：每条工作项不超过一句话

润色后更新 JSON 数据中的 `main_task` 字段，保持其他字段不变。

### 第三步：生成 docx

```bash
python3 <skill_dir>/scripts/generate_docx.py /tmp/daily_report_data.json
```

脚本会在 `~/Desktop/aispeech/日报/` 目录生成 `今日日报MMDD.docx`。

### 第四步：输出预览

生成后，直接读取 docx 文件内容，原样输出给用户预览。

**注意**：
- **不要自己编排格式**，直接从 docx 中读取实际内容输出
- **不要加序号**，脚本已经加好了 `(1)(2)(3)...`，不要再套一层
- **不要改写内容**，docx 里写的是什么就输出什么
- **项目名行不带序号**，直接输出项目名（如 `skill_engine`），不要加 `(1)` 等序号
- 如果明日计划不足 3 条，不要输出预览，直接回到第二步/第三步重新生成

## 内容规范（硬性要求）

- **今日工作不得低于 5 行**：如果脚本采集到的有效工作项不足 5 条，必须从 `decisions`、`assistant_summaries`、`bash_commands` 中补充提炼，确保内容充实。宁可细化拆分，也不能留空
- **明日计划不得少于 3 行**：基于今日遗留问题、未完成任务、错误/issue、todo 进度等多维度生成，确保至少 3 条。可以包括：bug 修复计划、性能优化方向、功能迭代计划、技术调研安排等
- **需要的支持默认写"暂无"**：除非用户明确提出了外部依赖或资源需求，否则一律写"暂无"，不要从 errors/issues 中自行推断

## 输出格式要求

- 每个项目名单独一行，加粗显示
- 每条工作项一行，格式为 `(序号) {做了什么}，{效果/指标}`
- 有指标就带，没指标就说清楚做了什么、解决了什么问题
- 合并同类：同一功能的多次迭代合并为一条
- 简洁有力：每条工作项控制在一句以内，拒绝流水账
- 如果某个 session 没有实质性工作，直接省略
- 不要编造数据中没有的量化指标
- 用自然的工作语言，读起来像正常工程师写的日报，不要像营销文案
